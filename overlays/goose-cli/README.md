# Goose crate2nix packaging notes

This overlay packages Goose with **crate2nix** to get a highly cacheable,
per-crate Rust build graph, while still preserving Goose's custom V8 setup.

## File ownership: generated vs hand-maintained

### Generated

- `Cargo.nix`
  - generated by `crate2nix`
  - checked in on purpose; we do **not** generate it during the build
  - post-processed so it can use `rootSrc` instead of assuming the generated
    file lives next to the Goose workspace source

### Hand-maintained

- `default.nix`
  - source surgery for Goose + V8
  - crate overrides and compatibility shims
  - final package assembly (`goose-cli` + `goose-server`)
- `goose-workspace-nix.patch`
  - stable Goose workspace path rewrites that can live as checked-in patches
- `rusty-v8-nix.patch`
  - stable `rusty_v8` and Chromium build-file rewrites that can live as
    checked-in patches
- `crate-hashes.json`
  - crate2nix hashes for fetched git crates that are not on crates.io
- `hipstr-no-serde-bytes.patch`
  - transitive crate patch for `hipstr 0.6.0`
- `patch_allocator_build.py`
  - V8 helper patch used during source preparation
  - forces whole-archive allocator linkage and drops the obsolete
    `rust_allocator_no_nightly_capability` shim path for our
    `RUSTC_BOOTSTRAP`-enabled Nix rustc
- `patch_apple_toolchain_host_build_tools.py`
  - patches Chromium's separate Apple host-build-tools toolchain
  - keeps proc-macro / build-script / generator links off lld and off
    fatal-linker-warnings mode under our Darwin toolchain layout
- `normalize_cargo_nix.py`
  - post-processes freshly generated `Cargo.nix`
  - uses `nix-manipulator` to add `rootSrc` and rewrite local source paths via
    the parsed Nix AST instead of regex-based text replacement
- `updater.py`
  - updates only the upstream Goose source hash in `sources.json`
  - it deliberately does **not** regenerate `Cargo.nix`

## Why the package is complex

The complexity is coming from three places at once:

1. Goose is a large Rust workspace.
1. Goose vendors a custom V8 integration that has to be patched in-place.
1. crate2nix/buildRustCrate expose a few incompatibilities that Cargo normally
   papers over.

That means this package is not just "generate Cargo.nix and build". It also
needs:

- a patched Goose source tree
- a checked-in `Cargo.nix`
- a handful of crate-specific fixes
- a custom V8 toolchain layout on Darwin

## Current hand-maintained fixes

### Source preparation in `default.nix`

The package now splits source preparation into two derivations so normal Goose
workspace churn does not invalidate the expensive V8 source input.

- `patchedV8Src`
  - prepares the `rusty_v8` fork on its own derivation boundary
  - applies `rusty-v8-nix.patch` for the stable V8-side source edits
  - patches allocator rules, whole-archive handling, and the Apple
    host-build-tools toolchain
  - links in a separate `chromiumToolchainBundle` derivation that constructs
    the `third_party/rust-toolchain` and
    `third_party/llvm-build/Release+Asserts` layout expected by Chromium/V8
- `v8NativeDrv`
  - runs the actual GN/Ninja native V8 build in its own derivation
  - stores `librusty_v8.a` plus the generated `gn_out/project.json` metadata
  - gives Nix a stable cache boundary around the truly expensive part
- `patchedSrc`
  - prepares the Goose workspace and then copies in `patchedV8Src` under
    `vendor/v8-goose-src`
  - keeps Goose-only path rewrites, lockfile updates, and crate2nix inputs

`patchedSrc` still does the non-negotiable Goose-side surgery:

- fetch Goose from the release tag in `sources.json`
- fetch the Goose-specific `rusty_v8` fork from `overlays/goose-v8/sources.json`
- inject that fork under `vendor/v8-goose-src`
- copy the two web UI logo assets into `crates/goose-cli/static/img/`
- apply `goose-workspace-nix.patch` so `crates/goose-cli/src/commands/web.rs`
  reads those local logo files instead of walking out to
  `documentation/static/img`
- apply `goose-workspace-nix.patch` so `vendor/v8/Cargo.toml` uses the local
  path dependency
- rewrite the `v8-goose` entry in `Cargo.lock`

### Crate overrides

These are intentionally centralized in `crateOverrides` inside `default.nix`.

- `commonGooseOverride`
  - shared native/build inputs for the Goose workspace binaries
- `xcapLinuxOverride`
  - Linux-only `libxcb` dependency
- `hipstrOverride`
  - applies `hipstr-no-serde-bytes.patch`
  - fixes `hipstr 0.6.0` failing to resolve `serde_bytes`
- `rmcpOverride`
  - injects `CARGO_CRATE_NAME` and `CARGO_PKG_VERSION`
  - fixes `rmcp 0.12.0` using `env!()` for Cargo package vars
- `llamaCppSysOverride`
  - injects `CARGO_CFG_TARGET_FEATURE = ""`
  - adds `cmake` and `pkg-config`
  - fixes `llama-cpp-sys-2` expecting a Cargo env var that
    `buildRustCrate` does not currently export
- `v8GooseOverride`
  - points the `v8-goose` crate directly at `patchedV8Src`
  - points the crate at prebuilt native outputs from `v8NativeDrv`
  - sets `RUSTY_V8_ARCHIVE` to the checked native archive in the Nix store
  - sets `RUSTY_V8_PREBUILT_GN_OUT` so the build script regenerates bindings
    without rerunning GN/Ninja
  - avoids rerunning GN/Ninja during ordinary Goose crate rebuilds

## Why `Cargo.nix` is checked in

An earlier in-build / IFD-style `crate2nix generate` approach failed during
`cargo metadata` for Goose's git-based `llama-cpp-rs` workspace dependency.
Once materialized inside the build, that dependency no longer looked like a
complete Cargo workspace to `cargo metadata`.

Checking in `Cargo.nix` avoids that failure mode and keeps the actual package
build free of crate2nix-at-build-time complexity.

## Regenerating `Cargo.nix`

Fast path:

```bash
nix run .#nixcfg -- ci pipeline crate2nix --write --package goose-cli
```

Manual flow:

When the Goose version or lockfile changes enough that the Rust dependency graph
must be regenerated, use this flow from the repo root.

1. Update source hashes first.

   - `overlays/goose-cli/updater.py` updates the Goose source hash.
   - `overlays/goose-v8/sources.json` is maintained separately if the V8 fork
     revision changes.

1. Build the already-patched Goose source tree:

   ```bash
   nix build --impure --no-link --print-out-paths \
     .#goose-cli.passthru.patchedSrc
   ```

   Save the printed store path as `PATCHED_SRC`.

1. Run crate2nix against that patched source tree:

   ```bash
   crate2nix generate \
     -f "$PATCHED_SRC/Cargo.toml" \
     -o overlays/goose-cli/Cargo.nix \
     -h "$PATCHED_SRC/crate-hashes.json" \
     --default-features
   ```

1. Normalize the generated file for this repo layout:

   ```bash
   python overlays/goose-cli/normalize_cargo_nix.py \
     overlays/goose-cli/Cargo.nix
   ```

1. Rebuild and continue iterating on any crate overrides that are still needed.

## Update checklist

When touching this package, verify these questions explicitly:

- Did Goose itself change, or only the source hash?
- Did the custom V8 fork revision change?
- Does `Cargo.nix` need regeneration, or can it stay as-is?
- Did any override become unnecessary after the version bump?
- Did any new crate start depending on Cargo env vars that `buildRustCrate`
  does not provide?
- Did the V8 source build start using new toolchain assumptions on Darwin?

## Recommended validation

At minimum:

```bash
nix run .#nixcfg -- ci pipeline crate2nix --package goose-cli
nix build --impure -L -vv --show-trace .#goose-cli --no-link
```

That build now includes an install-check smoke test for the `goose` binary,
covering `goose --version` and a clean-home `goose info` invocation.

Then verify the installed CLI actually starts:

```bash
GOOSE=$(nix path-info .#goose-cli)/bin/goose
"$GOOSE" --version
HOME=$(mktemp -d) XDG_CONFIG_HOME=$(mktemp -d) XDG_STATE_HOME=$(mktemp -d) \
  "$GOOSE" info
```

For long builds, tee the output to a log file and run it in the background so
failures can be inspected without losing context.
